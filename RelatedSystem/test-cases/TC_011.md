## Test Case 11: Reminder1 SMS is not sent after CSV enrollment when service enabled is false (AU)

**Test Case ID:** SMS-CSV-REMIND1-NOSEND-001  
**Test Case Title:** Verify Reminder1 SMS is not scheduled or sent after CSV enrollment when service enabled is false (CSV registration, brand AU)  
**Priority:** High  
**Automation ID:** C00556  
**Grade:** D  
**Preconditions:**
- [Admin UI] **SMS Status:** Remind1 ON  
- [Admin UI] **Remind interval:** Remind1 after 1 day (or as per delayInMins configuration)  
- OWL/BISCUIT conditions for this test: **service enabled: false**, **brand: AU**. When service enabled is false, SMS will not be scheduled or delivered.
- CSV registration is used for enrollment; service enabled: false, brand: AU for the contract/MDN under test.
- Postman (or equivalent) is available; access to AWS (SQA/UAT), Titan DB and ISS DB for verification queries.

**Test Steps:**

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | In Postman, prepare an MDN that meets the OWL/BISCUIT conditions with **service enabled: false**, **brand: AU** (see Test Data for MDN requirements). | MDN/account is ready with ① "使い方サポート" contract, ② Date of birth, ③ gender, ④ brand AU, service enabled: false. |
| 2 | Send the GET request; search for **"smsconfig"** and scroll down. Verify **"isEnabled": false** and **"delayInMins"**: 0, 10, and 20 as per user expectation. | smsconfig shows isEnabled: false and delayInMins (0, 10, 20) as expected. |
| 3 | If **"delayInMins"** is not as per user expectation, set the expected delay time in the PUT request and execute the request. | Delay configuration is updated as required. |
| 4 | Create a CSV file containing **some** contract(s)/MDN(s) with **service enabled: false**, **brand: AU** and **some** contract(s)/MDN(s) with **service enabled: true**, **brand: AU** in the format required by the system for enrollment. *(Including both types allows differentiation: records for service enabled: true will appear in DB; records for service enabled: false will not—so we can confirm the file was processed and that absence of SMS is due to the flag.)* | CSV file is created with both types of records (service enabled: false and service enabled: true, brand AU) in the correct format. |
| 5 | Log in to AWS for the target environment (SQA or UAT). Navigate to the S3 bucket used for enrollment. | Logged in to AWS; correct environment and bucket are available. |
| 6 | Upload the CSV file to the configured S3 bucket path used for enrollment (under **"New Enrollment"** section). | CSV file is uploaded successfully. *(Note: CSV upload runs every 10 minutes.)* |
| 7 | In Titan DB, run: `select * from carrier_integration.carrier_enrollment order by created_datetime desc;` Check that the file entry for the upload appears. Optionally verify that records for **service enabled: true** MDNs from the same file appear; records for **service enabled: false** MDNs may not display. | File entry is present in Titan DB; upload is recorded. Differentiation between service enabled: true (records display) and service enabled: false (records may not display) can be confirmed. |
| 8 | Wait for the time when Reminder1 would otherwise be sent (e.g. per delayInMins). | Sufficient time allowed. |
| 9 | In ISS DB, run: `select * from sms_system.message order by created_datetime desc limit 100;` For the test MDN (**service enabled: false**), confirm **Reminder1** SMS is **not** present or not sent. For **service enabled: true** MDNs from the same CSV, confirm Reminder1 SMS is present (to validate that the file was processed and the difference is due to the flag). | No Reminder1 SMS is scheduled or sent for the test MDN (service enabled: false). Reminder1 SMS is present for service enabled: true MDNs from the same file, confirming the file was processed and Reminder1 is not sent only when service enabled is false. |

**Expected result (overall):** Reminder1 SMS will **not** be sent when service enabled is false (brand AU). Including both service enabled: true and false records in the same CSV allows validation that the file was processed and that only service enabled: false records do not receive Reminder1 SMS.

- **MDN creation requirements (when creating an MDN):**
  - ① "使い方サポート" (usage support) contract  
  - ② Date of birth  
  - ③ Gender  
  - ④ Brand

- **Verification queries (備考):**
  - Titan DB (uploaded file entry):  
    `select * from carrier_integration.carrier_enrollment order by created_datetime desc;`
  - ISS DB (Reminder1 SMS not sent / no records for test MDN):  
    `select * from sms_system.message order by created_datetime desc limit 100;`

**Postconditions:** Test MDN (service enabled: false) does not have Welcome or Reminder1 SMS scheduled/sent (as expected). DB rows can be used for traceability. No mandatory cleanup unless test data must be removed per project rules.

**Note:** CSV file is processed every 10 minutes; allow for this and for Reminder1 delay when checking. When service enabled is false, no Reminder1 SMS is sent. Using a mix of service enabled: true and false in the same CSV makes it easier to validate that the file was processed and that the absence of Reminder1 SMS is due to service enabled: false.
