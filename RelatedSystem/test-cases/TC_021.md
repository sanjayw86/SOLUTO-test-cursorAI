## Test Case 21: Manual delivery mode — Titan GET/PUT, mock queue, duplicate check, approve to ISS

**Test Case ID:** SMS-MANUAL-DELIVERY-TITAN-ISS-001  
**Test Case Title:** Verify manual delivery mode via Titan APIs; `iss_dispatch_history` lifecycle; duplicate check; approve re-delivery to ISS; restore normal mode  
**Priority:** High  
**Automation ID:** C10000
**Grade:** D

**Preconditions:**

- **Environment:** **SQA** (Titan base URL in **Test Data**; use UAT if your run is there).
- **Tools:** Postman (or similar), **Titan DB** access, **ISS DB** access.
- **API key:** Titan **`X-Api-Key`** from **secrets** / Postman environment — never paste real keys into shared documents.
- **Timing:** Set **smsconfig** **delayInMins** for messages **1–3** (Welcome / Reminder1 / Reminder2) so all three types can appear in one session (Confluence *UAT — How to test SMS using Postman*).
- **Layman path:** If manual mode is **already ON**, skip to **Phase 2** step **5**. If environment starts in **normal** mode, run **Phase 1** or start at **Phase 2** step **5** to turn manual **ON**.

**Test Steps:**

### Phase 1 — Optional: normal mode (manual delivery OFF)

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | **GET** `smsDeliveryWorker` (**Test Data**). | **200**; note `deliveryEnabled` and `isNormalOperationMode`. |
| 2 | **PUT** `isNormalOperationMode` with **`value`: `true`**, **`brand`**: **SHARED** (unless project differs). | **`result`: true**. |
| 3 | **GET** `smsDeliveryWorker` again. | **`isNormalOperationMode`: true** ⇒ manual delivery **OFF**. |
| 4 | After an approved SMS trigger (enrollment/CSV per project), check **ISS** `sms_system.message`. | SMS visible per **normal** operation (not held only in mock queue). |

### Phase 2 — Manual delivery ON and load queue

| Step | Action | Expected Result |
|------|--------|-----------------|
| 5 | **PUT** `isNormalOperationMode` with **`value`: `false`** (same endpoint as step 2). | **`result`: true**. |
| 6 | **GET** `smsDeliveryWorker`. | **`deliveryEnabled`: true** and **`isNormalOperationMode`: false** ⇒ manual **ON**. |
| 7 | Upload **CSV** / trigger ingestion per project (manual path). | Rows appear in **`carrier_integration.iss_dispatch_history`**. |
| 8 | In **Titan DB** run SQL query: `select * from carrier_integration.iss_dispatch_history order by created_datetime desc;` (limit as needed). | **`Process_Status`** = **`TO_BE_PROCESSED`** (or equivalent); **`is_approved`** = **false** / unchecked. |
| 9 | In **`iss_dispatch_history`**, confirm **Welcome**, **Reminder1**, **Reminder2** for test subscriber(s). | All three SMS types listed as expected. |
| 10 | **ISS DB:** pending sends should **not** match “already delivered in normal mode” until after approval (per product). | Behaviour matches **manual** mode. |

### Phase 3 — No duplicates before approve

| Step | Action | Expected Result |
|------|--------|-----------------|
| 11 | Check the duplicate SMS through **Postman**, **GET** `issMock/queue/status` (**Test Data**). | Healthy queue / no duplicate anomaly per API (capture JSON). |
| 12 | In **Titan DB** run the duplicate-check SQL on **`carrier_integration.iss_dispatch_history`** (**Test Data**). | **No rows** (no duplicate `user_id` + `sms_category` pairs). |
| 13 | Count rows vs enrollment (Welcome / R1 / R2 per MDN). | Full expected set; nothing missing. |

### Phase 4 — Approve and verify ISS

| Step | Action | Expected Result |
|------|--------|-----------------|
| 14 | **POST** `issMock/queue/approveAll` (**Test Data**). | **2xx** success. |
| 15 | In **Titan DB** re-query **`iss_dispatch_history`** (same `select * … order by created_datetime desc` pattern as step 8; **Test Data**). | **`Process_Status`** → **`DELIVERED`** (or equivalent); **`is_approved`** → **true**. |
| 16 | **ISS:** `select * from sms_system.message order by created_datetime desc limit 100;` | **Welcome**, **Reminder1**, **Reminder2** visible for test MDN(s). |
| 17 | End-to-end check vs enrollment/config. | **All SMS delivered using Manual delivery mode** (Confluence success criterion). |

### Phase 5 — Reset environment

| Step | Action | Expected Result |
|------|--------|-----------------|
| 18 | **PUT** `isNormalOperationMode` with **`value`: `true`**. | **`result`: true**. |
| 19 | **GET** `smsDeliveryWorker`. | **`isNormalOperationMode`: true** (manual **OFF**) unless next test requires manual ON. |

**Expected result (overall):** Manual ON/OFF is verifiable via **GET**; with manual **ON**, SMS **queue** in **`iss_dispatch_history`** until **approve**; **no duplicate** user+category pairs; after **approve**, **ISS** shows **all** expected **Welcome / Reminder1 / Reminder2**; **`isNormalOperationMode`** restored per project policy.

**Test Data:**

- **Base URL (SQA example):** `https://titan.soluto-kddi-sqa.jpnmob-acyan.npr.aws.asurion.net`  
- Replace **`<TITAN_SQA_API_KEY>`** with the real key from your secure store.

- **GET — SMS delivery worker settings**

```bash
curl --location 'https://titan.soluto-kddi-sqa.jpnmob-acyan.npr.aws.asurion.net/api/system/settings/smsDeliveryWorker' \
--header 'Accept: application/json' \
--header 'X-Api-Key: <TITAN_SQA_API_KEY>'
```

- **PUT — Normal operation mode ON (manual OFF):** `.../smsDeliveryWorker/isNormalOperationMode` — body `{"brand": "SHARED", "value": true}`

```bash
curl --location --request PUT 'https://titan.soluto-kddi-sqa.jpnmob-acyan.npr.aws.asurion.net/api/system/settings/smsDeliveryWorker/isNormalOperationMode' \
--header 'Content-Type: application/json' \
--header 'X-Api-Key: <TITAN_SQA_API_KEY>' \
--data '{
    "brand": "SHARED",
    "value": true
}'
```

- **PUT — Manual delivery ON (normal operation OFF):** same URL; body `{"brand": "SHARED", "value": false}`

- **GET — ISS mock queue / duplicate check**

```bash
curl --location 'https://titan.soluto-kddi-sqa.jpnmob-acyan.npr.aws.asurion.net/api/issMock/queue/status' \
--header 'Accept: application/json' \
--header 'X-Api-Key: <TITAN_SQA_API_KEY>'
```

- **SQL — list mock queue (newest first)**

```sql
select * from carrier_integration.iss_dispatch_history order by created_datetime desc;
```

- **SQL — duplicate detection (`iss_dispatch_history`)**

```sql
select
  idh.original_payload ->> 'userId' as user_id,
  idh.original_payload ->> 'smsCategory' as sms_category,
  count(*)
from
    carrier_integration.iss_dispatch_history idh
group by user_id, sms_category
having count(*) > 1;
```

- **POST — Approve all**

```bash
curl --location --request POST 'https://titan.soluto-kddi-sqa.jpnmob-acyan.npr.aws.asurion.net/api/issMock/queue/approveAll' \
--header 'Accept: application/json' \
--header 'X-Api-Key: <TITAN_SQA_API_KEY>'
```

- **ISS verification**

```sql
select * from sms_system.message order by created_datetime desc limit 100;
```

**Note:** **`isNormalOperationMode`**: **`true`** = normal (manual **OFF**); **`false`** = manual **ON**. Normal scheduling uses **`message_target`**; manual queue uses **`iss_dispatch_history`**. **`Process_Status`** / **`is_approved`** names may differ by schema — match Titan DDL. If **brand** is not **SHARED**, confirm with Dev. Pair business testers with someone who has Postman/DB access for layman-friendly runs. Skip **Phase 1** if you only need the manual path in a short session.

**Postconditions:**

- Titan **GET** shows **`isNormalOperationMode`** per project default (usually **true**).
- **`iss_dispatch_history`** cleaned up per project rules.
- Save API responses and timestamps for audit.
