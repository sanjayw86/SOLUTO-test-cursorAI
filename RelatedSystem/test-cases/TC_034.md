## Test Case 34: ARCH KFS chat — push notification reception

**Test Case ID:** ARCH-CHAT-KFS-NOTIFICATION-PUSH-001  
**Test Case Title:** Verify push notification is received when ARCH sends **KFS** message to user while app is closed  
**Priority:** High  
**Automation ID:** C00631  
**Grade:** D

**Role:** **Operator** (send-side from ARCH).

**Preconditions:**

- **User:** **au** / **Kosapo** or **Tsukasapo** test profile with **KFS** entry available (**Mypage → 破損・盗難・紛失 お手続き → チャットでご相談** or team-approved path). Confirm brand (AU / UQ) per build.  
- **Session:** **ARCH** can send to the **target user’s KFS chat session** (establish or resume session per **project setup** — e.g. user opened KFS chat earlier, then fully closed app before steps below).  
- **App state (at verification time):** App is **not launched** (closed / background-killed per **project policy**).  
- **Notification settings:**
  - **OFF:** App **in-app** notification switch (confirm exact label and location in product settings).
  - **ON:** **OS** push notification permission for the app.  
- **ARCH context:** **[故] Chat (KFS)** only for this test (not **[つか] Message (Soluto)**).  
- **Skill selection (ARCH):** Select **"[故] Chat (KFS)"**.  
- **Amazon Connect:** **KFS** routing profile assigned for the operator (wrong profile may route to **Soluto**).  
- **Browser:** **Chrome** for ARCH.  
- **Environment alignment:** One line end-to-end (app + ARCH + Connect if used). Do not mix SQA app with UAT ARCH.

**Test Steps:**

### Phase A — ARCH readiness (KFS)

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Open **ARCH** for **SQA** or **UAT** (match app). Select **Skills** as **"[故] Chat (KFS)"**. Sign in as **Operator**. Confirm **Routing profile** is team-approved **KFS** (Connect) if required by line. | Correct environment + **KFS** skill; Operator session ready. |

### Phase B — App / OS notification pre-check

| Step | Action | Expected Result |
|------|--------|-----------------|
| 2 | On device, verify app is **not launched**. | App is fully closed / not active. |
| 3 | Verify app-level **in-app** notification is **OFF**, and **OS** push for the app is **ON**. | Configuration matches precondition. |

### Phase C — Trigger and verify push notification

| Step | Action | Expected Result |
|------|--------|-----------------|
| 4 | From **ARCH**, send chat message(s) to the **target user’s KFS** session. | Message send succeeds from ARCH. |
| 5 | Observe device **without** launching the app. | **Push notification** appears in the **OS** notification area. |

---

## Expected result (overall)

When **ARCH** sends a **KFS** chat message to the user while the app is not launched, the device shows an **OS push notification**, with **in-app** notifications **OFF** and **OS push** **ON**.

---

## Test Data

- **User / MDN:** Kosapo or Tsukasapo test account with **KFS** enabled.  
- **Message text:** any short chat text from ARCH — **step 4**.  
- **Notification evidence:** screenshot of in-app vs OS toggles (optional, recommended).

---

## Note

- If no push: confirm app process fully stopped, **OS** permission, network, **ARCH/app tier** alignment, and that the session is still **KFS** (not **Soluto**).  
- If **Connect** routing is wrong, chats may not be **KFS** — fix profile and retry.  
- After verification, **close or release** the conversation on ARCH per team policy.

- ARCH conversation closed or handed off per operations policy.
